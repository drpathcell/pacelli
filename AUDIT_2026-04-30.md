# Pacelli — App audit, 2026-04-30

**Scope:** Full functional walkthrough on real iPhone (iOS 26.3.1) via Appium + WebDriverAgent. App build 1.0.0+7. User signed in via Apple. Audit Household created during this run.

**Method:** Autonomous UI walker hit Home → Tasks → Calendar → Settings and every settings sub-screen. Code-side cross-checks for findings flagged in UI. Screenshots in `.audit-2026-04-30/` (28 PNGs).

**TL;DR:** App architecture and security foundations are solid. UI has rough edges, the AI integration is sprawl that earns its keep poorly, and **the burn-all-my-data feature is broken for Apple Sign-In users** — a privacy compliance issue that must block any further user signups.

---

## 🔴 CRITICAL — must fix before another tester touches the app

### C1 · Burn All My Data is broken for Apple Sign-In users
**Severity:** privacy compliance + security. This is the only feature that lets a user revoke their data; for SIWA users it does nothing useful.

**Evidence (code-level):** `lib/features/settings/presentation/screens/burn_data_screen.dart` lines 235–253. The reauthentication switch handles `google.com` and email/password only. Apple Sign-In users have no email password, so `EmailAuthProvider.credential(...)` fails with `wrong-password` / `invalid-credential`, the burn aborts, and the user is told they got the password wrong (a password they never set).

**Fix:** add an `else if (providerIds.contains('apple.com'))` branch that calls back into `AppleSignInService` to reauthenticate, then proceeds. The same SIWA flow that worked tonight applies — get a fresh idToken/rawNonce/authorizationCode, build the OAuth credential, call `user.reauthenticateWithCredential(...)`. ~30 lines.

**Why this is P0:** Without it, Apple users cannot delete their data. That violates GDPR/CCPA "right to erasure" expectations and undermines the entire privacy story Pacelli markets. Cannot ship to App Store review with this broken — Apple reviewers test account deletion.

### C2 · Untranslated l10n keys are leaking to the user
**Where:** Capabilities screen (`Settings → What Can Pacelli Do?`). The user sees raw strings `capGroupFeedback` and `capGroupFeedbackDesc` instead of "Feedback & Insights".

**Evidence:** `lib/features/capabilities/presentation/screens/capabilities_screen.dart` `_resolve()` switch has 56 cases but is missing entries for `capGroupFeedback` and `capGroupFeedbackDesc`. The default fallback `return key;` exposes the raw l10n key. Translations themselves exist in all 3 ARB files — the bug is purely the missing switch case.

**Fix:** add 2 cases to the switch (5 lines). Better fix: replace the giant switch with a `Map<String, String Function(AppLocalizations)>` table to make this kind of drift impossible.

**Why this is P0:** Looks like the app shipped half-finished. First impression on the Capabilities marketing screen.

## 🟡 HIGH — embarrassing if shipped

### H1 · User profile shows "Hello, Friend" / "Unknown (You)" after Apple Sign-In
Apple only returns the user's name on the **first** authorization. On every subsequent sign-in (including ours, where the dev cert was revoked + reauthorized) Apple returns just the user identifier — empty `givenName` / `familyName`.

`apple_sign_in_service.dart` lines 60–71 only update `displayName` if Apple returned the name. After that initial chance is gone, you're stuck with an empty displayName — UI then falls back to "Friend" / "Unknown (You)".

**Fix:** persist displayName in Firestore profile doc on first sign-in. On subsequent sign-ins, hydrate the local user from that doc. Also fall back to email local-part as a last resort instead of "Friend".

**Why H1 not P0:** doesn't break functionality, but every Apple SIWA user looks like an anonymous ghost in the household member list.

### H2 · AI is a UX swamp and adds no daily value
The app currently has THREE separate AI surfaces:
1. **Center FAB** (always-on robot icon between Tasks and Calendar) — opens an in-app chat UI
2. **Settings → AI Assistant** — pick from Claude / Gemini / ChatGPT, paste API key, configure MCP
3. **Capabilities screen** — "AI" badges on features

Plus an `aiChat` Cloud Function backing it.

This is sprawl. The center FAB is permanently visible and pulls the eye. Configuring it is a 4-step process most users will never complete. The chat itself uses keyword-based intent routing (per `CLAUDE.md`), not real LLM tool-use, so it's worse than just typing into ChatGPT.

You said you want to **remove it and let external AI agents talk TO Pacelli via MCP** — that's the correct architecture. The MCP server (`mcp-server/`) already exists and exposes Pacelli's data through Model Context Protocol, which is what AI agents actually use.

**Fix (concrete removal list):**
- Delete `lib/features/ai_chat/` (entire directory)
- Delete `lib/features/settings/data/ai_assistant_service.dart`
- Delete `lib/features/settings/presentation/screens/ai_assistant_screen.dart`
- Remove center FAB from `MainShell` — bottom nav goes from 5 tabs back to 4 (Home, Tasks, Calendar, Settings)
- Remove route `/ai-chat`, `/ai-assistant` from `app_router.dart`
- Remove "AI Assistant" entry from settings root
- Remove `aiSupported` flag and "AI" badges from `capabilities_screen.dart`
- Remove `capGroupAi`, `capGroupAiDesc`, plus all `aiChat*` and `aiAssistant*` l10n keys (3 ARB files each)
- Remove the `aiChat` export in `functions/src/index.ts` and the underlying intent-routing logic
- Keep `mcp-server/` and the `pacelli://capabilities` MCP resource — those are how external AI agents talk to Pacelli

**Estimated work:** half a day. ~2,000 LOC + l10n entries removed. Net code delta is hugely negative.

### H3 · The AI FAB occludes other UI
On the Tasks empty state ("All caught up!") the floating AI robot icon visually competes with the bottom-right `+` FAB used to create tasks. Users instinctively tap the AI icon thinking it's the action button. Removing the AI FAB (per H2) fixes this incidentally.

## 🟢 MEDIUM — polish

### M1 · Privacy & Encryption screen is marketing copy only
The screen lists every encrypted field in detail. Users don't care; they need to know "your data is encrypted, end-to-end, period." Replace 60% of the screen with a single bold reassurance + a "View details" disclosure for the inventory of encrypted fields.

### M2 · Settings root has 11 first-class entries; cluster them
Currently flat. Group as:
- **Account** (User profile, Sign Out)
- **Household** (Manage household, Notifications)
- **App** (Appearance, Privacy & Encryption summary, About)
- **Help** (What Can Pacelli Do?, Feedback)
- **Danger Zone** (Burn All My Data — keep separate visually)

Drop the AI Assistant entry entirely (per H2).

### M3 · Empty states need clear primary actions
- **Tasks empty state** says "All caught up!" — no CTA. Add an outlined "Create your first task" button so users don't have to find the small `+`.
- **Calendar empty state** doesn't exist — calendar always shows the month grid. Fine, but the Plans/Checklists sections under it should disappear when zero of each, or show "Tap to create" affordances.

### M4 · About Pacelli screen is a stub
Just "View licenses" + Close. Add: version number (visible already on Settings root), build number, contact email (`juancarlos.celis92@gmail.com`?), privacy policy link, terms link, GitHub link if open-source intended.

### M5 · "Hello, Friend" + clock should be a real header
The home screen header is `"Hello, $name"` + date. With Apple SIWA + no captured name, this looks broken. Tied to H1; will resolve once H1 is fixed.

### H4 · Google Drive connect fails with `cloud_firestore/permission-denied` (added during build 10 testing)

**Reported:** Apple-signed-in user clicks `Settings → Household → File Storage → Connect Google Drive` and gets a snackbar `"Failed to connect Google Drive: [cloud_firestore/permission-denied]"` after the OAuth scope grant succeeds.

**Root cause:** The Firestore security rule for `household_drive_config/{householdId}` requires `isMember(householdId)`, which does an `exists()` lookup at the deterministic path `/household_members/{uid}_{householdId}`. If the user's member doc was created in the legacy random-UUID format (pre-rules-rewrite), `exists()` returns false and the write is rejected.

A `HouseholdService.migrateMemberDocIds()` exists, but it's fired-and-forgotten on splash (`unawaited(...)`), so it can race against any household-scoped Firestore write made within the first second or two after sign-in — including Drive connect.

**Fix (build 11):** `lib/features/household/presentation/screens/drive_setup_screen.dart`
- Added `_ensureMembership(uid)` helper that probes the deterministic doc path. If the doc is missing, it `await`s `migrateMemberDocIds()` and re-probes. Returns an actionable error if still missing.
- `_connectDrive()` now calls `_ensureMembership` BEFORE OAuth scope request — wasted scope grants on an unfixable membership state are no longer possible.
- `_checkExistingConfig()` triggers the migration eagerly on screen open so the `_isLoading` indicator covers it.
- Diagnostic `debugPrint` of uid + householdId on success path so any future drift is observable.

**Estimated work:** 30 min (done).

**Pending:** the same race exists for any other household-scoped write made immediately after sign-in. Long-term fix is to `await migrateMemberDocIds()` inside the splash redirect before navigating to home — safer than dotting the migration call across every entry point.

## 🔵 LOW — nice-to-have

### L1 · Notifications offers only 3 hardcoded reminder times
"At due time / 1 hour before / 1 day before". Real users want "30 min before" or "morning of" or per-task overrides. Either add per-task notification override at task creation, or extend the global list with a custom-time picker. Not blocking v1.

### L2 · Calendar tab unclear vs Tasks tab
Calendar shows tasks/plans/checklists for the day, but you can't *create* anything from there. Either:
- **Option A:** make Calendar truly read-only, frame it as "What's on today" — strip the action affordances
- **Option B:** allow tap-and-hold on a date to create a task scheduled for that day

### L3 · Plans feature is undiscovered
"New Plan" button in Calendar tab is the only way to create Plans. It's a meaningful feature (multi-day trip plans, meal plans) but buried. Consider promoting it to the bottom nav (replacing the AI FAB after removal — see H2).

### L4 · Submit Feedback flow is well-built but never used
You built `Feedback & Insights` with type/rating/message + history + weekly digests + Cloud Functions + MCP tools. That's serious infrastructure for a beta of n=1. Either get more testers in to use it, or quietly hide the Insights/Digests tabs until n>5 testers.

### L5 · Push notifications capability registered but no APNs key uploaded
You've enabled `Push Notifications` capability on the App ID, but I don't see an APNs auth key configured in Firebase Cloud Messaging. Until that's done, no remote pushes will reach the device — only local notifications work. Verify before App Store submission.

### L6 · 24 lint info-level findings remaining from `flutter analyze`
Mostly `prefer_const_constructors`. Not bugs but noise. A 1-hour pass would clean them up. Can be deferred.

## What's working well (don't change)

- **Encryption architecture** — AES-256-CBC with HKDF, fresh IVs, key wrapping per user. Solid.
- **Firestore security rules** — household_id denormalization on every doc, deterministic membership, default-deny. This is well done.
- **Dual backend (Firebase + SQLite)** — the abstraction layer is clean. Both implementations have parity.
- **Material 3 theming with 3 schemes (Pacelli sage / Claude purple / Gemini ocean)** — looks polished, good palette discipline.
- **Localisation** — 1,200+ keys synchronized across EN/ES/IT (despite the 2 missing switch cases).
- **CI/CD + Fastlane** — `bundle exec fastlane ios beta` is the entire ship pipeline. That's how it should be.
- **Burn flow batch ordering and Firestore cache clearing** — when this works (Google + email users), it's correct: deletes data in proper order, terminates and clears the offline cache to prevent stale-read after re-auth.
