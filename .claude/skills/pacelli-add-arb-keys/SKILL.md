# Pacelli — Add ARB Keys (Sync All Locales)

## Overview
This skill adds new internationalisation keys to ALL ARB locale files in the Pacelli app simultaneously, ensuring they never go out of sync. Use this whenever new UI strings are needed — whether for a new feature, a new screen, or a fix.

## Project Location
The Pacelli project is at the user's local path (typically `~/Developer/pacelli`). In Cowork sessions it is mounted at `/sessions/*/mnt/pacelli/`.

## Architecture
- **ARB directory**: `lib/l10n/`
- **Template (English)**: `lib/l10n/app_en.arb` — source of truth, contains `@key` metadata for parameterised strings
- **Supported locales**: `app_en.arb` (English), `app_es.arb` (Spanish), `app_it.arb` (Italian) — check the directory for any new locales added since this skill was written
- **Extension**: `lib/core/utils/extensions.dart` — provides `context.l10n.keyName` shorthand
- **Generated code**: `lib/l10n/app_localizations*.dart` — auto-generated by `flutter gen-l10n`

## Steps

### 1. Identify what keys are needed
Ask the user (or determine from context) what new UI strings are required. For each string, define:
- **Key name**: camelCase, prefixed by feature area (e.g., `homeWelcomeTitle`, `taskPriorityUrgent`, `settingsThemeMode`)
- **English value**: the default string
- **Whether it has placeholders**: e.g., `{count}`, `{name}`, `{error}` — these need `@key` metadata in the template

### 2. Discover all locale files
```bash
ls lib/l10n/app_*.arb
```
This will show all current locale files. Every new key must be added to ALL of them.

### 3. Add to the English template (`app_en.arb`)
- Add the key and English value before the closing `}`
- If the key has placeholders, also add the `@key` metadata entry immediately after:
```json
  "myNewKey": "Hello, {name}! You have {count} tasks.",
  "@myNewKey": {
    "placeholders": {
      "name": {"type": "String"},
      "count": {"type": "int"}
    }
  },
```
- For **plurals**, use ICU syntax:
```json
  "taskPendingAttachments": "{count} {count, plural, =1{file} other{files}} ready to upload",
  "@taskPendingAttachments": {
    "placeholders": {
      "count": {"type": "int"}
    }
  },
```
- Simple keys (no placeholders) need only the key-value pair — no `@key` entry

### 4. Add to ALL other locale files
For each non-English ARB file (`app_es.arb`, `app_it.arb`, etc.):
- Add the key with the translated value
- **DO NOT** add `@key` metadata entries — those belong only in the template
- **DO NOT** translate placeholders — `{name}`, `{count}`, etc. must remain exactly as-is
- Use natural, native-speaker-quality translations

### 5. Validate JSON
After editing, ensure all ARB files have:
- No trailing commas before `}`
- Proper escaping of quotes within values (`\"`)
- Valid JSON structure

### 7. Using the new keys in Dart code
In any widget that needs the new string:
```dart
import '../../../../core/utils/extensions.dart';

// Then in the build method:
Text(context.l10n.myNewKey)

// With placeholders:
Text(context.l10n.myNewKey(userName, taskCount))
```

### 8. Build
No code changes needed beyond using the key — Flutter auto-generates the Dart bindings:
```bash
flutter clean && flutter pub get && flutter run
```

## Common Pitfalls
- **Forgetting a locale file**: If `app_en.arb` has a key but `app_es.arb` doesn't, the build will fail with a clear error
- **Translating placeholders**: `{count}` must stay as `{count}` in ALL locales
- **Duplicate keys**: Each key must be unique across the file — search before adding
- **Trailing commas**: JSON doesn't allow trailing commas — the last entry before `}` must NOT have a comma
- **Missing `@key` metadata**: If a key uses placeholders but has no `@key` entry in `app_en.arb`, the generated code won't have typed parameters
